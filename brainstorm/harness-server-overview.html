Data flow

An action (button press / wait) flows down through the stack; observations (events, snapshots) flow back up. The session folder on disk is authoritative — server can be killed and resumed without losing a session.

Reading the diagram. Solid arrows = data going down (action / request). Dashed arrows = data coming back up (response / reload). Turn-based: the emulator only ticks during an action POST. Between calls, the game is frozen and the LLM has unbounded thinking time.

What the LLM can observe

Across the routes, the agent has access to every player-observable aspect of the game. The same data is available three ways: live during an action (events in the POST response), pulled on demand (snapshot via /state, screen via /screen, map via /map), or queried from history (/events, /snapshots/history, /search/*).

  • Dialogue and on-screen text — every text_display event carries the full decoded string (no need to read the tilemap).
  • Player position — x, y, facing direction, current map id.
  • Movement — player_moved, map_loaded, ledges, doors.
  • Party state — 6 slots × (species, level, hp_cur, hp_max, status, type1, type2).
  • Active battle — enemy species, level, current HP, types; player's active mon and its 4 moves.
  • Inventory — bag with up to 20 items, plus quantities.
  • Money and badges bitfield.
  • Pokedex — owned and seen bitfields (151 species each).
  • Event flags — game-progression bits (story checkpoints).
  • Menus — open/close events plus snapshot's text_box_id.
  • Map structure — static layout (warps, signs, NPC spawn points) for any of 248 maps; live NPC positions on current map.
  • Screen pixels — 160×144 PNG snapshot at any frame.
  • History queries — past events filtered by id/category/frame range; past snapshots; full-text search over dialogue.
  • Self-managed memory — append-only journal.md survives across context resets.

Route catalog

All routes follow the GET-as-docs / POST-as-execute pattern. Hitting GET /<route> returns a JSON doc with params, returns, and an example. Hitting POST executes.

What's intentionally not here

To keep the harness minimal and observation-based, the current server does not expose:

  • A narrative renderer (events are raw with decoded text fields; profile-based "naive vs aware" rewriting is a follow-on).
  • Pathfinding (/walk_to(x,y)) — agent should explore.
  • Pokemon-specific helpers (/catch, /heal) — agent figures these out by observation.
  • Cross-session "canonical" save points (current saves are session-scoped; benchmark presets are a planned addition).
  • Authentication — V1 is localhost only.
Save/load UX gaps worth knowing. /save and /load work, but there is no /saves/list route yet — the slot list is only exposed via the 404 body of a failed load. There is also no per-save metadata (note, frame, timestamp) and no delete endpoint. These are easy to add when the agent starts using saves heavily.
